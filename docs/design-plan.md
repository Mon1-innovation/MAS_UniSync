# MAS UniSync Design Plan

## Goal

MAS UniSync provides cross-device synchronization for Monika After Story's Ren'Py
`persistent` data. It keeps one active client session per profile key, uploads persistent
changes to a central server, and lets users manage profile keys from a web
portal authenticated through the Monika Love Flarum forum.

Only `persistent` is synchronized. Normal Ren'Py save slots are out of scope.

## Confirmed Decisions

- Client code must be delivered only as a MAS/Ren'Py submod.
- No sidecar executable, local daemon, or external client process.
- Sync is enabled only after the user enters a profile key. Once a key is
  configured, network, key validation, or lock acquisition failure should show
  an error by raising an exception.
- The profile key is generated by the central website.
- Users cannot provide custom profile keys.
- Profile keys are stored in plaintext by the server and can be viewed again.
- Profile keys identify sync profiles. They are not used to log in to the
  center website.
- The center website login uses Flarum authentication.
- Users paste the profile key manually in a submod setting panel.
- The center server URL is also stored through MAS API key storage. A production
  default value will be provided after the service is live.
- Server-side backups use a 10-day daily rotation, one backup per day.
- The submod does not create local backup snapshots before applying cloud
  persistent data.
- Conflict policy is cloud-first.
- The server can inspect metadata and admins can download persistent files; no
  end-to-end encryption is required.
- Flarum admin/backstage groups map to the center service `admin` role.
- Admins get an `Admin` tab in the center website.
- Flarum avatars are used in the center website.

## System Overview

The system has two deployable areas:

1. Central server
   - Web portal for Flarum login and profile key management.
   - REST API for the submod.
   - Lock service for single-active-client enforcement.
   - Persistent version storage and daily backup rotation.
   - Admin panel and audit logging.

2. MAS submod
   - Setting panel for server URL and profile key.
   - Button/link to open the center website profile key page.
   - Startup lock acquisition.
   - Persistent download before gameplay when cloud data is newer.
   - Hook around Ren'Py persistent saving.
   - Asynchronous upload after local persistent save.
   - Strict persistent type validation before upload.

## Central Server

### Recommended Stack

- API/Web: FastAPI or Go HTTP service.
- Database: PostgreSQL.
- Object storage: local Docker volume for MVP, with a compatible abstraction for
  MinIO/S3 later.
- Deployment: Docker Compose.

### Flarum Authentication

The center website logs users in through Flarum:

1. User enters forum credentials on the center website login page.
2. Server calls Flarum `/api/token` with `identification` and `password`.
3. If Flarum returns a token and user id, login is considered successful.
4. Server fetches user profile data, including username, display name, avatar,
   and groups.
5. Server creates or updates its local user record.
6. Server creates its own web session for the center website.

Flarum token handling:

- The server may keep the token only for the session or refresh user metadata at
  login time.
- The Flarum password is never stored.
- The Flarum token is not exposed to the submod.

### Roles

The center service has two roles:

- `user`
- `admin`

On every web login, the server checks the user's Flarum groups. If the user is
in a configured admin/backstage group, the local role is set to `admin`.
Otherwise it is set to `user`.

Group matching must be configurable:

```text
ADMIN_FLARUM_GROUP_IDS=1,5
ADMIN_FLARUM_GROUP_NAMES=Admin,后台
```

Roles are refreshed at login so removing a user from the Flarum admin group also
removes center admin access on the next login.

### Profile Keys

Profile keys are generated on the center website by logged-in users.

Properties:

- Randomly generated by the server.
- Not user-customizable.
- Stored in plaintext.
- Visible repeatedly in the user's Profile Keys page.
- Used by the submod as the profile identity.

Suggested format:

```text
maspk_<random-32-bytes-base64url-or-hex>
```

Profile key refresh:

- A logged-in user can refresh a profile key.
- The profile record keeps the same internal `profile_id`.
- The old key is invalid immediately.
- Current persistent and backups remain attached to the same profile.
- The user must paste the refreshed key into the submod.

Although profile keys are not center website login credentials, they authorize
submod access to a profile's sync data. Anyone with a valid key can attempt to
lock, upload, or download that profile through the submod API.

### Locking

The lock is scoped to one profile key/profile, not to the entire Flarum account.
The server does not track or distinguish concrete devices. It only cares that a
client request presents a valid profile key and, for lock-protected operations,
the current lease token.

Rules:

- Only one client session may hold a profile lock at a time.
- Lock TTL: 60 seconds.
- Heartbeat interval: 15 seconds.
- If a client crashes or loses network, another client can acquire the lock
  after the TTL expires.
- Admins can force-release a lock from the Admin tab.

Startup behavior:

- No network: raise exception.
- Missing key: sync is disabled and MAS startup continues.
- Invalid key: raise exception.
- Banned key/profile/user: raise exception.
- Lock already held: raise exception.

### Persistent Storage

The server stores uploaded persistent files as opaque binary objects.

The server does not unpickle or execute Ren'Py persistent contents.

For each profile:

- `current` points to the latest successful upload.
- `daily_backups` keeps one backup per day for the latest 10 days.
- Multiple uploads on the same day replace that day's daily backup with the
  latest successful upload for that day.

Metadata stored per version:

- version id
- profile id
- object path
- sha256
- size
- Ren'Py version
- MAS version if available
- upload timestamp

### Admin Tab

Users with role `admin` see an Admin tab after logging in to the center website.

Admin user list:

- Flarum avatar
- username/display name
- Flarum user id
- local role
- profile count
- last website login
- last submod use
- last upload time
- current lock status
- ban status
- storage usage

Admin profile detail:

- view profile metadata
- download current persistent
- download daily backups
- restore current persistent from a daily backup
- view recent activity
- view current lock status and expiry time
- force-release lock
- ban/unban user, profile, or key
- refresh/revoke profile key

Admin downloads and moderation actions must be audit logged.

### Bans

Ban targets:

- user
- profile
- profile key

Default ban behavior:

- Block lock acquisition.
- Block upload.
- Block download.
- Allow admins to view and download data.

Submod handling:

- Server returns HTTP 403 with a machine-readable ban reason.
- Submod raises an exception and displays the reason.

### Audit Logging

Audit log required for:

- admin viewing user details
- admin downloading current persistent
- admin downloading backup persistent
- admin restoring backup persistent
- admin banning or unbanning
- admin force-releasing lock
- profile key creation
- profile key refresh
- profile key revocation

Audit fields:

- actor user id
- actor role
- action
- target user id
- target profile id
- target key id if applicable
- IP address
- user agent
- timestamp

## Submod Client

### Compatibility Targets

- Ren'Py 6.99 / Python 2
- Ren'Py 8.2.3 / Python 3

Both Ren'Py 6.99 and Ren'Py 8.2.3 SDKs are available under:

```text
j:\Renpy
```

The exact SDK directory names should be confirmed before writing version-specific
test commands.

### Cross-Version Persistent Compatibility

Ren'Py 8 persistent files can contain pickle references to:

```text
renpy.revertable RevertableList
renpy.revertable RevertableDict
renpy.revertable RevertableSet
```

Ren'Py 6.99 does not provide the `renpy.revertable` module. In an isolated
Ren'Py 6.99 CLI reproduction with a Ren'Py 8 persistent, the first load failure
was:

```text
ImportError: No module named revertable
```

The compatibility layer must therefore provide a minimal `renpy.revertable`
module on Ren'Py 6.99 before persistent loading reaches `cPickle.loads`.

The minimal compatibility module can alias Ren'Py 6's existing revertable
classes:

```python
from renpy.python import RevertableDict, RevertableList, RevertableSet
```

This is not intended to fully implement Ren'Py 8 rollback behavior. It only
allows Ren'Py 6.99 to unpickle Ren'Py 8 persistent values and then re-save them
through Ren'Py 6's own persistent writer.

After loading a Ren'Py 8 persistent in Ren'Py 6.99, the conversion path should
delete these compatibility-sensitive keys before saving the Ren'Py 6 persistent:

```renpy
del persistent._voice_mute
del persistent._mas_acs_pre_list
del persistent._mas_windowreacts_notif_filters
```

Observed reasons:

- `_voice_mute` is legacy Ren'Py/DDLC voice state and is treated as unimportant
  by MAS import logic. Ren'Py 6 can recreate it from `renpy/common/00voice.rpy`.
- `_mas_acs_pre_list` is a pre-accessory name list. Deleting it lets the target
  MAS runtime recreate the field with its own list type/defaults.
- `_mas_windowreacts_notif_filters` is a window reaction notification filter
  dict. Deleting it lets the target MAS runtime recreate the field with its own
  dict type/defaults.

These deletions are not a substitute for the `renpy.revertable` compatibility
module. The module fixes the first unpickle failure; the key cleanup reduces
runtime and type compatibility risk after the load succeeds.

Cross-version conversion verification should use a temporary save directory:

```powershell
$env:APPDATA = "<temp-dir>"
& "J:\Renpy\renpy-6.99.12.4-sdk\renpy.exe" "<renpy6-project-dir>" --lint
```

The test must confirm:

- the Ren'Py 8 persistent zlib payload decompresses
- Ren'Py 6.99 unpickles without `ImportError`
- important MAS data such as player name, sessions, affection, and event
  databases survive conversion
- the re-saved persistent uses `renpy.python.Revertable*` references rather than
  `renpy.revertable.Revertable*`
- the three cleanup keys are either absent immediately before save or recreated
  by Ren'Py 6/MAS with Ren'Py 6-compatible types

### HTTP Client

The submod must use standard library networking:

- Python 2: `urllib2`, `ssl`, `json`, `Queue`
- Python 3: `urllib.request`, `ssl`, `json`, `queue`

No `requests` dependency should be required for MVP.

If TLS, DNS, timeout, or connection setup fails, the client treats it as network
failure and raises an exception at startup or on required sync operations.

### Setting Panel

The submod adds a MAS setting panel by registering itself through the MAS Submod
API. The registration pattern should follow:

```text
e:\GithubKu\MAICA_MTTSubmod\game\Submods\MAICA_MttsSubmod\header.rpy
```

The relevant pattern is:

```renpy
init -990 python:
    store.mas_submod_utils.Submod(
        author="...",
        name="MAS UniSync",
        description=_("..."),
        version=mas_unisync_version,
        settings_pane="mas_unisync_settingpane"
    )
```

Then the settings UI is implemented as:

```renpy
screen mas_unisync_settingpane():
    ...
```

The panel should include:

- Center server URL
- Profile key input
- Open Profile Keys Page button
- Test Connection button
- Sync status
- Last successful upload time
- Last successful download time
- Current lock state

The Open Profile Keys Page button opens:

```text
<server_url>/account/profile-keys
```

The profile key should be stored in MAS's API key storage when available,
following the MAICA MTTS submod pattern:

```renpy
store.mas_api_keys.api_keys.update({"MAS_UniSync_Profile_Key": profile_key})
store.mas_api_keys.save_keys()
```

The center server URL should use the same storage path:

```renpy
store.mas_api_keys.api_keys.update({"MAS_UniSync_Server_URL": server_url})
store.mas_api_keys.save_keys()
```

Reads should use the MAS helper when available:

```renpy
store.mas_getAPIKey("MAS_UniSync_Profile_Key")
store.mas_getAPIKey("MAS_UniSync_Server_URL")
```

Non-secret UI state may be stored in `persistent` or a small sync config file,
but the profile key and server URL should use the MAS API key mechanism for
consistency with existing submods.

Sync is considered enabled as soon as a non-empty profile key is saved. Saving a
profile key from the setting panel should immediately run one sync attempt and
acquire the profile lock.

The panel should guide the user to:

1. Open the center website.
2. Log in with the Flarum account.
3. Generate a profile key.
4. Paste the profile key into the submod panel.
5. Test connection.

### Startup Flow

1. Load local sync config.
2. If profile key is missing, skip sync and allow MAS startup.
3. Call `/v1/me` or `/v1/profile/resolve` with the profile key.
4. If key is invalid, raise exception.
5. If user/profile/key is banned, raise exception.
6. Acquire profile lock.
7. If the profile key is already locked, raise exception.
8. Fetch cloud current persistent metadata.
9. Compare remote sha256 with local persistent sha256.
10. If remote is newer, download remote persistent and either load it into
    Ren'Py memory during startup or replace the local persistent file on the
    non-memory path.
11. If local and remote conflict without a safe shared baseline, cloud wins:
    download and use the cloud persistent.
12. If local is newer or unsynced and no conflict is detected, enqueue upload.
13. Start heartbeat thread.
14. Allow MAS startup to continue.

### Save Hook

The submod hooks persistent saving.

Preferred compatibility strategy:

- Wrap `renpy.persistent.save()`.
- Call the original function first.
- After it succeeds, read the final local persistent file from the Ren'Py save
  directory.
- Calculate sha256.
- If the hash changed, enqueue an asynchronous upload.

Reasoning:

- Ren'Py 6.99 and 8.2 both expose `renpy.persistent.save()`.
- Ren'Py 8.2 adds persistent signing, so the client should upload Ren'Py's final
  written bytes instead of reserializing persistent itself.

Upload behavior:

- Upload runs in a background thread.
- The background thread must not mutate Ren'Py store objects.
- Upload only receives file bytes, metadata, server URL, and profile key.
- Duplicate hashes are skipped.
- If upload fails during normal gameplay, the failure is logged silently and
  stored in local sync status.
- If the final quit upload fails, the submod raises an exception.
- The quit-upload exception message must remind the user to manually back up the
  local persistent save before troubleshooting or using another client.

### Quit Upload Hook

For normal MAS exits, the submod may also register a function plugin on `_quit`
through the MAS Submod API. MAS provides this mechanism in `zz_submods.rpy` via
`store.mas_submod_utils.registerFunction` and the `functionplugin` decorator.

Reference examples:

```text
J:\MAS\MonikaModDev-PC\Monika After Story\game\zz_submods.rpy
E:\GithubKu\MAICA_MTTSubmod\game\Submods\MAICA_MttsSubmod\cache_remover.rpy
```

Recommended pattern:

```renpy
init python:
    @store.mas_submod_utils.functionplugin("_quit", priority=-100)
    def mas_unisync_on_quit():
        ...
```

The `_quit` hook should be optional and should not replace the persistent save
hook. Its job is to improve normal-exit behavior:

- force a final persistent save if safe
- enqueue or perform a final upload attempt
- flush a short upload wait window if an upload is already pending
- release the server lock
- stop the heartbeat thread

The `_quit` hook should keep a strict timeout so quitting MAS cannot hang
indefinitely on network I/O. Crashes, force-kills, and network loss are still
handled by lock TTL expiration.

If the final upload attempt fails during `_quit`, the hook should raise an
exception with a message that tells the user the cloud sync failed and that they
should manually back up their local persistent before continuing.

### Local Persistent Replacement

When the non-memory sync path applies a cloud version, the submod writes the
downloaded bytes to `persistent.unisync-new` and then renames that temporary
file over the local `persistent`. It does not create or rotate local backup
snapshots. Server-side daily backups remain the recovery mechanism for uploaded
persistent versions.

### Persistent Type Guard

Before uploading, the submod validates `persistent.__dict__` recursively.

Allowed value types:

- `None`
- string/unicode
- bool
- int/long/float
- list/tuple/set/frozenset
- dict
- Ren'Py `RevertableList`
- Ren'Py `RevertableDict`
- Ren'Py `RevertableSet`
- `datetime.date`
- `datetime.datetime` with no timezone
- `datetime.time` with no timezone
- `datetime.timedelta`

Disallowed:

- arbitrary custom class instances
- `store.*` custom classes
- objects that cannot be safely compared or serialized
- recursive structures that exceed recursion depth

MAS already has a similar strict persistence check in `event-handler.rpy`; the
submod should reuse the same design approach where possible.

## Verification Plan

### Server

- Flarum login succeeds with valid credentials.
- Flarum login fails with invalid credentials.
- Flarum admin group maps to local `admin`.
- Non-admin Flarum user maps to local `user`.
- Avatar URL is imported and displayed.
- Profile key generation creates a random non-custom key.
- Profile key is visible again after page reload.
- Profile key refresh invalidates the old key and preserves profile data.
- Lock acquisition blocks a second active client using the same profile key.
- Lock expires after TTL if heartbeat stops.
- Upload updates current persistent.
- Multiple uploads in one day produce one daily backup for that day.
- Daily backups retain only the latest 10 days.
- User can restore current persistent from a daily backup.
- Admin can restore a user's current persistent from a daily backup.
- Admin can view and download another user's current persistent.
- Admin download creates an audit log entry.
- Ban blocks submod lock/download/upload.
- Admin can force-release lock.

### Submod

- Ren'Py 6.99 can load the submod without syntax errors.
- Ren'Py 8.2.3 can load the submod without syntax errors.
- Missing key leaves sync disabled and allows startup.
- Invalid key raises a clear exception.
- No network raises a clear exception.
- Existing active lock on the same profile key raises a clear exception.
- Valid key and available lock allow startup.
- Remote newer persistent downloads before gameplay.
- Local save triggers asynchronous upload.
- Duplicate hash does not upload repeatedly.
- Disallowed custom class in persistent blocks upload.
